# v1.19.0 — Freshness Context and Matchday Trust Signals

## Purpose

v1.19.0 makes the existing stored-data freshness model easier to interpret during real matchdays.

The release follows a read-only audit of the deployed v1.18.1 runtime. That audit established that a successful provider sync and a stale dashboard snapshot are compatible facts: the sync result describes the latest attempt, while freshness describes the age of the last successful stored snapshot.

The release does not change sync cadence, provider configuration, runtime environment, or stored fixture data.

## What changed

### Schedule-aware freshness context

`GET /fixtures/sync/status` now includes an additive `freshness_context` object. The same object is available at:

```text
GET /live-match-centre
data_freshness.freshness_context
```

The object can expose:

```text
state
schedule_timezone
last_success_at
last_success_at_local
next_scheduled_run_at
snapshot_becomes_stale_at
snapshot_becomes_stale_at_local
stale_before_next_scheduled_run
diagnostic
message
```

The values are derived from already stored sync timestamps, configured freshness thresholds, and read-only schedule metadata. Stored provider timestamps remain unchanged.

### Truthful diagnostics

The release distinguishes:

| Situation | Meaning |
|---|---|
| `latest_sync_failed` | The latest provider refresh failed; the dashboard is showing the last successful stored snapshot. |
| `snapshot_stale_before_next_scheduled_refresh` | The latest refresh succeeded, but the stored snapshot is stale before the next configured slot. |
| `snapshot_will_be_stale_before_next_scheduled_refresh` | The latest refresh succeeded; its stored snapshot will cross the stale boundary before the next configured slot. |
| `snapshot_within_freshness_window` | The latest successful stored snapshot is currently within its freshness boundary. |

A schedule-aware diagnostic does not trigger a sync or predict that the next refresh will succeed.

### Dashboard trust signals

The dashboard shows a Freshness Context summary for:

- last successful stored snapshot;
- next scheduled refresh;
- stale-after boundary;
- the factual diagnostic.

When freshness is stale or the latest provider refresh failed, the Live Match Centre uses:

```text
Last confirmed live from stored snapshot
```

It also explicitly states that current live status is not inferred from a fixture kickoff time.

## Boundaries preserved

- The fixed provider schedule remains `03:45`, `09:45`, and `12:45` in `Asia/Singapore`.
- Freshness thresholds remain unchanged.
- No provider request, scheduled-sync trigger, browser polling, Telegram delivery, database write, Docker change, Cloudflared change, Ollama change, or active runtime `.env` change is introduced.
- Time is never used to infer a live or completed fixture.
- The v1.18.1 contract remains: an `unknown` fixture can display as scheduled only when a valid timezone-aware kickoff is future and scores are absent; at/past-kickoff unknown fixtures remain unavailable.
