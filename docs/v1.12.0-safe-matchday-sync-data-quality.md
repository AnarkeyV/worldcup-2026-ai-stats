# v1.12.0 — Safe Matchday Sync, Audit History, and Data Freshness

## Purpose

v1.12.0 improves the project’s operational safety for real World Cup matchdays without pretending that provider data is more complete or more current than it is.

The release adds a persisted audit trail for fixture sync attempts, optional safe scheduling, independent control for sync-generated completed-match Telegram alerts, and clearer dashboard data-freshness states.

## What Changed

### Persisted fixture sync audit history

Fixture sync runs are now stored in the database rather than existing only in process memory. Each stored run records safe operational information such as:

- source and provider
- trigger type
- start and completion time
- terminal outcome
- fetched, created, updated, and newly-completed counts
- a bounded and redacted error summary when a run fails

The latest status and recent history can be read without triggering a provider request.

### Explicitly opt-in scheduling

The provider fixture scheduler is disabled by default.

When enabled intentionally, it:

- waits a full configured interval before its first provider sync
- performs fixture sync only
- prevents overlapping scheduled runs
- does not fetch rich match details
- does not send Telegram alerts

### Safer Telegram policy

Completed-match Telegram alerts generated by sync are controlled separately and remain disabled by default.

The existing Telegram test action remains an explicit operator action. A configured bot token and chat ID alone do not enable sync-generated alerts.

### Factual dashboard freshness states

The dashboard now communicates whether stored fixture data is:

- not yet synced
- fresh
- aging
- stale
- unavailable
- affected by a latest-sync failure

Stored match detail also exposes its local refresh timestamp. This reflects when the current stored provider payload was last updated; it does not claim that a live provider request has just happened.

## Configuration

The following `.env` settings are safe defaults:

```dotenv
PROVIDER_SYNC_SCHEDULER_ENABLED=false
PROVIDER_SYNC_INTERVAL_MINUTES=30
FIXTURE_SYNC_FRESH_AFTER_MINUTES=60
FIXTURE_SYNC_STALE_AFTER_MINUTES=180
TELEGRAM_COMPLETED_MATCH_ALERTS_ENABLED=false
```

Do not enable scheduling or sync-generated Telegram alerts unless you deliberately want those side effects on the Windows Docker runtime.

## Deliberate Non-Goals

This release does not add:

- automatic rich-detail backfill
- automatic Telegram alerts from the scheduler
- event-specific alerting
- provider-event deduplication or correction history
- inferred assists or player analytics
- factual match-report export

Those features require a separate provider-contract review and test-backed design so that the dashboard never fabricates football data.

## Windows Runtime Verification After Release

After the GitHub release is published, verify on the Windows Docker host:

1. Pull the tagged `main` release.
2. Rebuild and start Docker Compose.
3. Confirm `/health` reports version `1.12.0`.
4. Open the dashboard and confirm the Sync panel renders a truthful no-sync or latest-persisted-run state.
5. Keep scheduler and completed-match Telegram alerts disabled unless intentionally testing them.

No provider sync, rich-detail backfill, or Telegram message is required for baseline runtime verification.
