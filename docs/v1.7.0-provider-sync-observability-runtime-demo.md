# v1.7.0 — Provider Sync Observability & Runtime Demo

## Purpose

This guide documents the **v1.7.0 — Provider Sync Observability & Runtime Demo** milestone for the World Cup 2026 AI Stats project.

The milestone improves the project’s runtime demo story by making fixture sync activity visible across:

- FastAPI API endpoints
- Static dashboard UI
- Prometheus metrics
- Grafana dashboards
- Docker Compose runtime checks

The goal is to make provider/sample sync behavior easier to explain during a portfolio review, interview, or live technical walkthrough.

## What v1.7.0 Adds

### Backend Runtime Status

A new runtime status endpoint is available:

```text
GET /fixtures/sync/status
```

This endpoint shows the latest fixture sync state.

It reports:

- sync status
- source
- provider
- last run time
- last successful run time
- duration
- fetched fixture count
- created fixture count
- updated fixture count
- newly completed fixture count
- newly completed fixture IDs
- last error message, if any

Example response before any sync has happened:

```json
{
  "status": "not_started",
  "source": null,
  "provider": null,
  "last_run_at": null,
  "last_success_at": null,
  "duration_seconds": null,
  "total_fixtures": 0,
  "created": 0,
  "updated": 0,
  "newly_completed_count": 0,
  "newly_completed": [],
  "last_error": null
}
```

Example response after sample fixture sync:

```json
{
  "status": "success",
  "source": "sample",
  "provider": "sample_data",
  "last_run_at": "2026-06-18T00:00:00+00:00",
  "last_success_at": "2026-06-18T00:00:00+00:00",
  "duration_seconds": 0.012,
  "total_fixtures": 4,
  "created": 4,
  "updated": 0,
  "newly_completed_count": 4,
  "newly_completed": [
    "sample-mex-rsa-2026-06-11",
    "sample-usa-par-2026-06-12",
    "sample-fra-sen-2026-06-16",
    "sample-arg-dza-2026-06-16"
  ],
  "last_error": null
}
```

## New Prometheus Metrics

v1.7.0 keeps the existing fixture sync metrics:

```text
worldcup_fixture_sync_runs_total
worldcup_fixture_sync_created_total
worldcup_fixture_sync_updated_total
worldcup_fixture_sync_newly_completed_total
```

It also adds new sync observability metrics:

```text
worldcup_fixture_sync_duration_seconds
worldcup_fixture_sync_fetched_total
worldcup_fixture_sync_last_run_timestamp_seconds
worldcup_fixture_sync_last_success_timestamp_seconds
```

These metrics make it easier to show:

- when sync last ran
- whether sync succeeded or failed
- how long sync took
- how many fixtures were fetched
- how many records were created or updated
- whether any newly completed fixtures were detected

## Dashboard Runtime Panel

The FastAPI static dashboard now includes a new panel:

```text
Provider Sync Runtime
```

URL:

```text
http://localhost:8000/dashboard
```

The panel displays:

- sync status badge
- provider
- last run time
- last success time
- sync duration
- fixtures fetched
- fixtures created
- fixtures updated
- newly completed count
- last sync error

This gives the project a stronger visual runtime demo without needing to explain only raw API responses or Prometheus metrics.

Dashboard title:

New provider sync panels include:

- Fixture Sync Runs
- Last Successful Fixture Sync

These panels are backed by Prometheus queries against the `worldcup_fixture_sync_*` metrics.

## 11. Optional Provider Sync Demo

Only run this if API-Football credentials are configured in `.env`.

Check `.env`:

```powershell
type .env
```

Required values:

```text
API_FOOTBALL_KEY=...
API_FOOTBALL_BASE_URL=https://v3.football.api-sports.io
API_FOOTBALL_WORLD_CUP_LEAGUE_ID=1
API_FOOTBALL_SEASON=2026
```

Run provider sync:

```powershell
curl -X POST http://localhost:8000/fixtures/sync/provider
```

If the key is missing or still set to a placeholder, this endpoint should safely return an error instead of crashing.

Expected missing-key behavior:

```json
{
  "detail": "API_FOOTBALL_KEY is not configured."
}
```

Then check:

```powershell
curl http://localhost:8000/fixtures/sync/status
```

This should show an error status with the last error message.

---

## Notes

- Runtime sync status is intentionally in-memory for this milestone.
- It resets when the backend container restarts.
- This is acceptable for a local runtime demo.
- A future milestone could persist sync run history in PostgreSQL.
- API-Football credentials remain optional and environment-driven.
- No secrets should be committed to the repository.
