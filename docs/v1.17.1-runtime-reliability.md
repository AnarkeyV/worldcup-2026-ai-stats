# v1.17.1 — Runtime Reliability Safeguards

## Release Summary

v1.17.1 strengthens the operational boundary around the self-hosted Windows runtime. It keeps useful Docker/container recovery while making Cloudflared and Ollama inspection-first, rather than treating them as services that project scripts should automatically restart or reconfigure.

The release is deliberately operator-focused. It does not change fixture data, match storytelling, scheduled sync, Telegram behaviour, the public dashboard layout, or the local model itself.

## What It Adds

### Read-only runtime status checker

`scripts/windows/get-worldcup-runtime-status.ps1` provides one local Windows check for:

- project directory and Docker engine availability
- Docker Compose container state
- local backend and dashboard health
- Cloudflared service state and public health
- local and public application version consistency
- host Ollama API and installed model names
- application-level local AI health
- the user-level `WorldCup2026 - Ollama Local AI` task state and last-run metadata

The script uses `PASS`, `WARN`, `FAIL`, and `INFO` output. `-FailOnCritical` returns a non-zero process result only when a critical local prerequisite such as the project directory, Docker engine, or backend health is unavailable.

### Safer automation boundary

The existing Windows startup and watchdog scripts retain their Docker responsibilities:

```text
start Docker Desktop when needed
bring the Compose runtime up
recover unhealthy Docker containers
```

They now treat these components as report-only:

```text
Cloudflared service and public reachability
Ollama host/API availability
application AI health
```

A failing public route or unavailable local AI is logged with a clear warning. The project scripts do not start, restart, stop, kill, reconfigure, create, reinstall, or download anything for Cloudflared or Ollama.

### Version consistency visibility

The status checker compares the local backend `/health` version with the public `/health` version. A mismatch becomes a warning rather than a silent false-positive public health result.

This specifically protects the local-first deployment model where a version value can be overridden by the active `.env`. The source release changes only safe defaults; updating an active runtime `APP_VERSION` remains an approved deployment operation with backup and verification.

### Recovery guidance

`docs/windows-runtime-recovery.md` records the safe recovery path for Docker, Cloudflared, and Ollama incidents. It intentionally keeps these out of Git and out of diagnostic output:

- active `.env` contents
- provider keys and Telegram credentials
- Cloudflare tunnel tokens, certificate files, credential JSON, and config contents
- Scheduled Task XML
- Ollama model files
- runtime logs and database data

The source changes were also executed as a read-only candidate check on the Windows runtime. The result showed healthy Docker containers, matching local/public v1.17.0 application versions, a running Cloudflared service, available `llama3.2:1b`, available application AI, zero critical failures, and zero warnings.

## Runtime Boundaries

- No status check starts or restarts Docker, performs a provider sync, changes the scheduler, sends Telegram, modifies Cloudflared, modifies Ollama, changes a Scheduled Task, or exposes configuration values.
- No active `.env`, database volume, model file, tunnel configuration, or runtime log is changed during source preparation.
- Deployment remains separately approved: candidate validation, non-secret configuration compatibility, timestamped backup for approved configuration changes, compose verification, and post-deployment local/public checks.

## Follow-up

The next user-facing feature milestone should be a live match centre and data-freshness improvement. It should remain grounded in stored provider data, make delayed/incomplete states obvious, and keep provider refresh work separate from normal dashboard reads.
